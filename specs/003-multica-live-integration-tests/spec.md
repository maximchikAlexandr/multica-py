# Спецификация функции: Live-интеграционное тестирование `multica-py`

**Feature ID:** `003-multica-live-integration-tests`  
**Статус:** Ready for implementation  
**Дата:** 2026-07-18

## 1. Обзор

Необходимо добавить в `multica-py` отдельный набор live-интеграционных тестов, проверяющий реальную цепочку взаимодействия:

`Python SDK → фактический Multica CLI → запущенный Multica backend → хранилище данных → ответ CLI → Python-модель или Python-исключение`.

Существующие проверки актуальности API и компонентные тесты остаются самостоятельными слоями качества. Новая функция должна обнаруживать несовместимости, которые невозможно надёжно выявить только статическим сопоставлением команд, подменой CLI или тестированием отдельных компонентов.

## 2. Цели

- Подтверждать, что публичные методы `multica-py` работают с реальным CLI и реальным экземпляром Multica.
- Выявлять несовместимости аргументов команд, форматов ответов, авторизации, контекста workspace и обработки ошибок.
- Давать быстрый блокирующий сигнал для критичных регрессий в pull request.
- Давать расширенный периодический сигнал о совместимости с актуальной или целевой версией Multica.
- Обеспечивать воспроизводимый локальный запуск и полезную диагностику при падении.

## 3. Не входит в область функции

- Повторное тестирование всей внутренней бизнес-логики backend Multica.
- Повторение полного браузерного E2E-набора Multica.
- Проверка визуального интерфейса.
- Полное исполнение всех команд CLI в каждом pull request.
- Проверка внешних AI-провайдеров, сторонних git-хостингов и иных необязательных внешних интеграций в базовом smoke-наборе.
- Нагрузочное и длительное soak-тестирование в первой версии функции.

## 4. Пользователи и заинтересованные стороны

- **Разработчик `multica-py`** — запускает тесты локально перед отправкой изменений.
- **Ревьюер pull request** — получает подтверждение, что изменение не нарушило базовую интеграцию.
- **Мейнтейнер релиза** — проверяет совместимость SDK с целевой версией Multica.
- **Мейнтейнер Multica** — получает воспроизводимый отчёт о несовместимости CLI/backend и Python SDK.

## 5. Пользовательские сценарии и тестирование

### US-001 — Быстрая проверка интеграции в pull request

**Приоритет:** P1  
**Как** разработчик `multica-py`, **я хочу** запустить компактный live-набор против изолированного экземпляра Multica, **чтобы** до слияния обнаружить нарушение базовой цепочки SDK–CLI–backend.

#### Критерии приёмки

1. **Дано** совместимое окружение и целевая версия Multica, **когда** запускается PR smoke-набор, **тогда** он подтверждает авторизацию, чтение workspace и как минимум один полный CRUD-сценарий.
2. **Дано** нарушение построения CLI-аргументов или декодирования ответа, **когда** выполняется затронутый smoke-тест, **тогда** тест завершается ошибкой и указывает проверяемую операцию.
3. **Дано** неготовое или недоступное окружение, **когда** начинается тестовая сессия, **тогда** тесты не продолжают выполнение как будто сервис готов, а возвращают диагностируемую ошибку подготовки окружения.

### US-002 — Проверка round-trip основных ресурсов

**Приоритет:** P1  
**Как** мейнтейнер SDK, **я хочу** проверять создание, чтение, изменение, перечисление и удаление основных сущностей, **чтобы** убедиться, что данные не теряются и модели SDK соответствуют реальным ответам.

#### Критерии приёмки

1. Для простого ресурса выполняется полный цикл create → get → list → update → delete.
2. Для составного сценария issue проверяются связи с project, labels и comments.
3. Текстовые значения с Unicode, переносами строк, кавычками, обратными слешами и пробелами сохраняются без искажения.
4. После удаления сущность больше не доступна через поддерживаемые операции чтения.

### US-003 — Проверка семантики частичных обновлений

**Приоритет:** P1  
**Как** пользователь Python API, **я хочу** различать непереданное поле, явное `null`-значение и пустое значение, **чтобы** частичное обновление не изменяло несвязанные данные.

#### Критерии приёмки

1. Поле с sentinel `Unset` (не передано) сохраняет прежнее значение на backend.
2. Пустая строка для `project.description` через SDK записывается как `""` и отличается от omit.
3. Явный JSON `null` clear для `description` проверяется oracle HTTP (`PUT` с `"description": null`); через CLI/SDK `None` в MVP запрещён (`ValidationError`), потому что upstream CLI не умеет эмитить JSON null.
4. Пустая коллекция проверяется на issue labels: после detach всех labels oracle `GET /api/issues/{id}/labels` возвращает `[]`.
5. Обновление одного поля не изменяет другие поля сущности.
6. Точная матрица: `contracts/presence-and-exceptions.md`.

### US-004 — Проверка ошибок и безопасности диагностики

**Приоритет:** P1  
**Как** пользователь SDK, **я хочу** получать стабильные Python-исключения для типовых ошибок реального Multica, **чтобы** приложение могло надёжно реагировать на них.

#### Критерии приёмки

1. Неверная авторизация → `AuthenticationError`; access collapse (resource A via client B) и missing resource → `NotFoundError`; validation → `ValidationError` per `contracts/presence-and-exceptions.md`.
2. Недоступность backend / mid-kill → `NetworkError`; unclassified nonzero CLI exit → `CommandExecutionError`.
3. Диагностика сохраняет этап/операцию без секретов.
4. Токены, секреты и полный набор переменных окружения не попадают в сообщение исключения, stdout, stderr или опубликованные диагностические материалы.

### US-005 — Изоляция workspace и тестовых запусков

**Приоритет:** P1  
**Как** разработчик, **я хочу** иметь изолированные тестовые профили, workspace и данные, **чтобы** тесты не зависели от моей локальной конфигурации и не влияли друг на друга.

#### Критерии приёмки

1. Клиент одного workspace не видит данные, созданные только в другом workspace.
2. Тестовый запуск не читает и не изменяет пользовательский профиль Multica по умолчанию.
3. Два независимых запуска не используют одинаковые имена или идентификаторы тестовых ресурсов без явного намерения.
4. Завершение одного клиента или сценария не нарушает работу другого изолированного клиента.

### US-006 — Периодическая расширенная проверка совместимости

**Приоритет:** P2  
**Как** мейнтейнер релиза, **я хочу** регулярно запускать расширенный набор против выбранной версии или ветки Multica, **чтобы** заранее обнаруживать schema drift и поведенческие изменения.

#### Критерии приёмки

1. Расширенный набор проверяет пагинацию, фильтры, nullable/optional-поля, timestamps, attachments и дополнительные read-only ресурсы.
2. Версия CLI и версия backend, используемые в одном запуске, однозначно определены в отчёте.
3. При проверке новой upstream-версии результат можно сравнить с последним успешным запуском целевой поддерживаемой версии.
4. Падение содержит достаточно информации для определения ресурса, операции и несовместимой версии.

### US-007 — Независимая проверка результата

**Приоритет:** P2  
**Как** мейнтейнер SDK, **я хочу** в критичных сценариях подтверждать состояние независимым интерфейсом, **чтобы** одинаковая ошибка сериализации и десериализации внутри SDK не скрыла дефект.

#### Критерии приёмки

1. Для выбранных операций данные, созданные через SDK, проверяются независимым чтением фактического состояния.
2. Для выбранных partial-update операций исходное состояние подготавливается независимо от проверяемого метода SDK.
3. Независимая проверка подтверждает сохранение непереданных полей и точность специальных текстовых значений.

## 6. Функциональные требования

- **FR-001:** Система тестирования ДОЛЖНА содержать отдельный live-набор, явно отличимый от unit-, component- и contract-тестов.
- **FR-002:** Live-набор ДОЛЖЕН использовать фактический исполняемый файл Multica CLI, а не тестовую подмену.
- **FR-003:** Live-набор ДОЛЖЕН выполнять операции против реально запущенного экземпляра Multica с постоянным хранилищем на время тестовой сессии.
- **FR-004:** Подготовка тестового пользователя, авторизации и workspace ДОЛЖНА выполняться без ручного браузерного взаимодействия.
- **FR-005:** Перед первым тестом ДОЛЖНА быть подтверждена готовность Multica принимать запросы.
- **FR-006:** Все тестовые CLI-профили, токены, workspace и пользовательские данные ДОЛЖНЫ быть изолированы от конфигурации разработчика.
- **FR-007:** Версии CLI и backend в воспроизводимом запуске ДОЛЖНЫ быть зафиксированы или однозначно определены.
- **FR-008:** Базовый smoke-набор ДОЛЖЕН вызывать `MulticaClient.workspaces.list()` и подтверждать, что primary workspace id из bootstrap присутствует в результате.
- **FR-009:** Базовый smoke-набор ДОЛЖЕН включать полный CRUD round-trip для ресурса **labels** (create/get/list/update/delete).
- **FR-010:** Базовый smoke-набор ДОЛЖЕН включать составной workflow issue с project, labels и comment на pinned target; отсутствие capability → setup/test failure (skip запрещён).
- **FR-011:** Тестовые данные ДОЛЖНЫ включать Unicode, многострочный текст, кавычки, обратные слеши и пробелы в именах или значениях.
- **FR-012:** Live-набор ДОЛЖЕН проверять матрицу presence для `project.description` и empty label collection по `contracts/presence-and-exceptions.md` (после wiring `Unset` и CLI `--title`).
- **FR-013:** Live-набор ДОЛЖЕН проверять, что изменение одного поля не меняет непереданные поля (`Unset`).
- **FR-014:** Live-набор ДОЛЖЕН проверять mapping реальных ошибок в точные публичные типы из `contracts/presence-and-exceptions.md`.
- **FR-015:** Live-набор ДОЛЖЕН отдельно проверять недоступность backend (`NetworkError`) и ненулевой код завершения CLI (`CommandExecutionError`).
- **FR-016:** Сообщения об ошибках и диагностические материалы НЕ ДОЛЖНЫ раскрывать токены, секреты или полный набор переменных окружения.
- **FR-017:** Live-набор ДОЛЖЕН проверять изоляцию данных между двумя workspace.
- **FR-018:** Тесты НЕ ДОЛЖНЫ зависеть от существующего пользовательского профиля Multica или изменять его.
- **FR-019:** После каждого запуска ДОЛЖНА выполняться layered cleanup: registry deletes → Compose `down -v --remove-orphans` (если environment Compose-managed) → postcondition audit по `contracts/live-test-interface.md` §9.
- **FR-020:** Очистка ДОЛЖНА выполняться и после успеха, и после падения; ошибка cleanup после pass валит session, после fail сохраняет primary failure и отдельно репортит cleanup failure.
- **FR-021:** При падении ДОЛЖНЫ сохраняться сведения о проверяемой операции, версии CLI, версии backend и релевантные обезличенные логи сервисов.
- **FR-022:** Разработчик ДОЛЖЕН иметь возможность запустить live-набор отдельно от быстрых локальных тестов.
- **FR-023:** CI ДОЛЖЕН разделять компактный блокирующий PR smoke-набор и расширенный периодический набор.
- **FR-024:** Блокирующий PR smoke-набор НЕ ДОЛЖЕН требовать frontend, браузер, почтовую доставку или внешнего AI-провайдера.
- **FR-025:** Расширенный набор ДОЛЖЕН проверять пагинацию и комбинации фильтров хотя бы для одного списочного ресурса.
- **FR-026:** Расширенный набор ДОЛЖЕН проверять корректность optional/null/empty-значений и временных меток в реальных ответах.
- **FR-027:** Расширенный набор ДОЛЖЕН проверять attachments на pinned target: upload/metadata/download/SHA-256/delete для файла ровно 1024 байт плюс zero-byte и Unicode/spaced filenames; отсутствие capability → fail, не skip.
- **FR-028:** Для критичных partial-update и round-trip сценариев ДОЛЖНА использоваться независимая проверка фактического состояния, не основанная исключительно на том же методе SDK.
- **FR-029:** Имена тестовых ресурсов ДОЛЖНЫ позволять определить принадлежность к конкретному запуску и избегать случайных коллизий.
- **FR-030:** Повторный запуск набора на неизменном окружении ДОЛЖЕН давать одинаковый функциональный результат и не зависеть от порядка предыдущих запусков.

## 7. Нефункциональные требования

- **NFR-001 — Воспроизводимость:** локальный и CI-запуск с одинаковыми версиями компонентов должен проверять одинаковые сценарии.
- **NFR-002 — Диагностируемость:** падение должно быть локализуемо до этапа подготовки окружения, Python-операции, CLI-вызова или ответа backend.
- **NFR-003 — Надёжность:** тесты не должны полагаться на фиксированные задержки как единственный способ определения готовности сервиса.
- **NFR-004 — Изоляция:** тестовые процессы не должны использовать общие изменяемые данные без явной синхронизации.
- **NFR-005 — Безопасность:** чувствительные данные должны маскироваться во всех пользовательских и CI-выводах.
- **NFR-006 — Скорость:** обязательный PR smoke-набор ДОЛЖЕН укладываться в SC-003: env startup ≤ 180s, tests ≤ 120s, total wall ≤ 300s на стандартном CI runner при уже доступных артефактах target.
- **NFR-007 — Расширяемость:** добавление нового ресурса в live-набор не должно требовать копирования полного lifecycle подготовки окружения.

## 8. Ключевые сущности

### Test Environment

Изолированный запущенный экземпляр Multica и связанное хранилище, существующие в течение тестовой сессии.

### Test Identity

Тестовый пользователь и его учётные данные, созданные или подготовленные без ручного участия.

### Test Workspace

Workspace, в котором создаются тестовые ресурсы. Для проверки изоляции используются как минимум два независимых workspace.

### CLI Profile

Изолированная конфигурация CLI с адресом сервиса, токеном и активным workspace, не связанная с пользовательским профилем разработчика.

### Test Resource

Созданная тестом сущность Multica с уникальным признаком принадлежности к запуску.

### Compatibility Target

Однозначно определённое сочетание версий Python SDK, CLI и backend, для которого сформирован результат тестирования.

### Diagnostic Bundle

Обезличенный набор сведений о падении: операция, версии, коды завершения и релевантные логи без секретов.

## 9. Граничные случаи

- Backend-процесс запущен, но ещё не готов принимать запросы.
- CLI доступен, но имеет несовместимую версию или формат вывода.
- Токен отсутствует, просрочен или не имеет доступа к выбранному workspace.
- Workspace указан в профиле, но удалён или недоступен.
- Ресурс удалён между list и get.
- Поле отсутствует в JSON, равно `null`, пустой строке, пустому списку или пустому объекту.
- Сервер возвращает дополнительные неизвестные поля.
- Сервер возвращает временную метку с другой точностью или смещением timezone.
- Текст содержит Unicode, emoji, переносы строк, кавычки, обратные слеши и ведущие/замыкающие пробелы.
- CLI пишет диагностические сообщения в stdout вместе с JSON — **MVP out of scope** (extended follow-up); harness treats decode failure as `JsonOutputError`/`ProtocolError` if encountered incidentally.
- CLI завершается ненулевым кодом без структурированного ответа — covered by T051.
- Backend становится недоступен во время операции — covered by T050 → `NetworkError`.
- Очистка отдельного ресурса невозможна из-за частично завершённого workflow — layered cleanup §9 continues to compose destroy.
- Два тестовых запуска выполняются параллельно — **unsupported in v1** (no xdist).
- Пагинация empty page / repeated cursor — **extended follow-up** beyond T059 happy path (12 issues / page_size 10).
- Бинарный attachment имеет нулевой размер, пробелы или Unicode в имени — covered by T063 on pinned target.

## 10. Критерии успеха

- **SC-001:** Базовый smoke-набор проверяет всю цепочку Python SDK → реальный CLI → реальный backend минимум для чтения workspace, одного CRUD-ресурса и одного составного workflow.
- **SC-002:** Намеренное нарушение имени CLI-флага, формата обязательного поля ответа или преобразования not-found ошибки приводит к падению соответствующего live-теста (MVP automation: `scripts/run_live_tests.py` mutation check / T071).
- **SC-003:** Обязательный PR smoke-набор: env startup ≤ 180s, tests ≤ 120s, total ≤ 300s на стандартном CI-runner при уже доступных артефактах целевой версии.
- **SC-004:** Десять последовательных запусков `pytest -m live_smoke tests/live` на неизменном pinned target завершаются без flaky failure (0 retries; flaky = любой non-deterministic fail across the 10 runs); MVP: T075.
- **SC-005:** После успеха или fail отсутствуют Compose containers/volumes проекта, temp HOME и profile path; допускается только явно сохранённый обезличенный diagnostic bundle.
- **SC-006:** Для каждого падения можно определить compatibility target, этап, ресурс и операцию без доступа к секретам.
- **SC-007:** Проверка двух workspace подтверждает отсутствие видимости тестовой сущности из несвязанного workspace.
- **SC-008:** Partial-update тесты проходят матрицу `contracts/presence-and-exceptions.md` case IDs `P-OMIT`, `P-EMPTY`, `P-SET`, `C-EMPTY`.
- **SC-009:** Специальные текстовые данные после round-trip byte-identical исходному Python `str` (Unicode/emoji/newlines/quotes/backslashes/spaces) для label name/color и issue description на pinned target.
- **SC-010:** Ни один опубликованный лог или текст исключения не содержит используемый тестовый токен в открытом виде.

## 11. Предположения (hard prerequisites — unmet → setup failure)

- Pinned target `v0.3.35` supports noninteractive development auth with `MULTICA_DEV_VERIFICATION_CODE=888888` when `APP_ENV=development`.
- Pinned target exposes workspace, labels, projects, issues, comments, attachments APIs used in contracts.
- Compose file `docker-compose.selfhost.yml` provides services `postgres` and `backend`; `/readyz` returns the body pinned in `contracts/bootstrap-http.md`.
- CLI and backend of one run come from the same release tag/`upstream_commit`.
- Base scenarios need no frontend/daemon/email/AI provider.
- Existing contract/component tests remain and are not replaced by the live suite.
- Parallel pytest workers (xdist) are unsupported in v1.

## 12. Зависимости и риски

- Изменения механизма development-authentication в Multica могут потребовать обновления подготовки окружения.
- Отсутствие стабильного способа определить версию backend ухудшит диагностику compatibility target.
- Часть ресурсов может требовать дополнительных сервисов и должна оставаться вне PR smoke-набора.
- Недетерминированная фоновая обработка Multica может потребовать явных условий ожидания состояния.
- Расширенный набор против upstream `main` может сигнализировать о ещё не выпущенных изменениях и не должен автоматически трактоваться как дефект поддерживаемого release.

## 13. Трассировка минимального релиза (MVP)
