# Research — Spec 005

Каждая запись содержит одно окончательное решение. Раздел «Rejected» фиксирует исторически рассмотренные варианты и не является списком разрешённых реализаций.

## R-001. Blocking agent execution

**Decision:** использовать real Multica backend/CLI/daemon и deterministic OpenCode-compatible executable.

**Rationale:** это проверяет orchestration, runtime registration, task dispatch, local-directory routing и filesystem side effect без сети, квот и stochastic model output.

**Rejected:** mock transport/backend; real cloud LLM in PR; direct file edit from test body.

## R-002. Runtime lifecycle

**Decision:** runtime создаётся только как следствие `multica daemon start --foreground` и удаляется остановкой daemon с ожиданием deregistration.

**Rationale:** Multica daemon auto-detects agent CLIs and owns runtime registration/heartbeat lifecycle.

**Rejected:** server-side runtime create/delete; direct database insertion.

## R-003. Project resource

**Decision:** использовать `local_directory` bound to daemon ID.

**Rationale:** требование состоит в изменении заранее созданного файла в локальной директории. `local_directory` запускает agent непосредственно там.

**Rejected:** `github_repo` worktree; copying files into daemon workspace.

## R-004. SDK boundary

**Decision:** добавить typed `client.projects.resources` and issue `project_id` support.

**Rationale:** live feature обязана проверять публичный SDK. Raw CLI inside test would обходить проверяемый продукт.

**Rejected:** subprocess raw CLI for resource attach; private transport calls; bootstrap API for project resources.

## R-005. Workspace provisioning

**Decision:** создать/удалить workspace существующим `BootstrapApiClient`.

**Rationale:** current public workspace SDK is not the provisioning boundary, while live bootstrap already owns isolated identity/workspace setup.

**Rejected:** expand public workspace CRUD; reuse shared workspace; direct SQL.

## R-006. Agent cleanup

**Decision:** archive agent, then verify non-routable.

**Rationale:** SDK exposes archive semantics; physical deletion is not required to eliminate routing risk.

**Rejected:** invent agent delete; leave active agent; delete through database.

## R-007. Fake OpenCode protocol

**Decision:** parse one `MULTICA_TEST_ACTION=<compact-json>` line from final prompt, enforce schema 1, perform atomic exact replacement and emit OpenCode JSONL events.

**Rationale:** machine-readable instruction is deterministic while still passing through real daemon prompt construction and provider process boundary.

**Rejected:** free-form natural-language parser; environment-only action that ignores issue content; successful empty process.

## R-008. Unexpected files

**Decision:** allow mutations for `target.txt`, `AGENTS.md`, `.multica/**`, `.opencode/**`, and `.agent_context/**`; require `control.txt` unchanged.

**Rationale:** Multica intentionally writes provider instructions, OpenCode skill discovery trees, issue context, and project resource metadata into the local directory.

**Rejected:** assert entire directory byte-identical except target; ignore every dotfile; allow arbitrary generated files.

## R-009. Test data generation

**Decision:** explicit deterministic builders and literal fixtures.

**Rationale:** CRUD/live payloads encode exact presence, Unicode, routing and cleanup intent; random factories obscure that intent.

**Rejected:** Polyfactory; Hypothesis; Faker-driven payloads.

## R-010. Parameterized execution

**Decision:** standard pytest parametrization with stable IDs.

**Rationale:** each case remains independently collectable, rerunnable, distributable and compatible with `--last-failed`.

**Rejected:** pytest-subtests; one loop containing all cases.

## R-011. Contract fixtures

**Decision:** ordinary reviewable JSON/TOML fixtures plus explicit semantic assertions.

**Rationale:** contract data is also consumed by scripts and requires independent inspection.

**Rejected:** Syrupy snapshots; expected data generated by production code during test.

## R-012. Docker lifecycle

**Decision:** preserve existing ComposeLifecycle and add independent CI final cleanup.

**Rationale:** project-specific target, diagnostics and bootstrap logic remains necessary; `if: always()` cleanup handles pytest process interruption better than another wrapper.

**Rejected:** testcontainers-python; fixture teardown as the only cleanup.

## R-013. Mutation testing

**Decision:** weekly/manual targeted mutmut run, separate dependency group, no live tests and no PR required check.

**Rationale:** mutation analysis reveals weak assertions in critical pure logic but is too expensive/noisy for every PR and irrelevant for Docker/LLM flows.

**Rejected:** mutmut over entire repository; mutation score 100%; no mutation audit.

## R-014. Real OpenCode

**Decision:** one scheduled/manual canary with externally supplied path/model/credentials, 15-minute timeout and 0.10 USD ceiling.

**Rationale:** real CLI compatibility remains observable without making external availability a merge dependency.

**Rejected:** mandatory DeepSeek free tier; provider auto-selection; multi-model matrix.

## R-015. Traceability

**Decision:** do not create Markdown matrix or capabilities manifest. Use stable pytest IDs, markers, contract tests and CI summaries.

**Rationale:** manual/generated documentation does not improve execution correctness and creates synchronization work.

**Rejected:** committed generated Markdown; custom requirement decorators.
